# Preprocess Probe Notes (2026-06-29)

Purpose: explain the images generated by `preprocess_probe.py` and why the tested preprocessing variants fail or help. This is an offline bench note only; it does not change the router or main app.

## Files

- OCR-input variants: `D:\LocalTranslateHub\.codex-run\manga-ocr-bench\rois\preprocess_probe\`
- OCR result table: `D:\LocalTranslateHub\.codex-run\manga-ocr-bench\rois\preprocess_probe\results.tsv`
- CV-only masks: `D:\LocalTranslateHub\.codex-run\manga-ocr-bench\rois\preprocess_probe\cv_masks\`

Variant meanings:

- `raw`: the original column crop from `columnize()`.
- `gray3`: grayscale crop converted back to 3 channels.
- `clahe`: local contrast enhanced grayscale.
- `mask_bw`: chosen text mask rendered as black ink on white background.
- `mask_bw_dilate`: `mask_bw` with one small dilation.
- `orig_on_white`: original crop pixels kept only where the chosen mask has ink; all background whitened.
- CV masks: `gray` = current Otsu polarity black-hat/top-hat, `lab` = Lab local color delta, `edge` = Canny-like edge mask, `grad` = morphology gradient mask.

## Main Result

Do not feed `mask_bw` / binarized masks to manga-ocr as the normal input. The model is much better on natural crops.

Average over the focused probe:

| variant | avg jp_ratio | read |
|---|---:|---|
| raw | 0.84 | best baseline |
| gray3 | 0.84 | roughly equal to raw |
| clahe | 0.84 | roughly equal to raw |
| orig_on_white | 0.68 | loses context, can hallucinate |
| mask_bw | 0.46 | often becomes dotted garbage |
| mask_bw_dilate | 0.46 | same failure |

Conclusion: preprocessing should be used for cheap CV mask/layout/crop selection, not as the default OCR image.

Implementation follow-up: see `D:\LocalTranslateHub\.codex-run\manga-ocr-bench\GEOMETRY-ROUTER-PLAN.md`. That file turns this probe result into the bench contract: masks serve geometry only; manga-ocr reads raw crops.

## Case Features and Failures

### purple-vert(42s)

Corrected ROI: `(500,460,700,850)`, two vertical columns. The natural crop has colored diamond background but enough luminance structure for the current gray/Otsu branch.

OCR:

- `raw`, `gray3`, `clahe`: readable (`散々ワガマっ`, `語っとい`).
- `mask_bw`, `mask_bw_dilate`: collapses to `．．．` style output.
- `orig_on_white`: one column hallucinated `ジャケットで`, the other becomes dots.

Why it fails: binarization removes anti-aliasing, stroke texture, background context, and subtle glyph shape cues. Manga-ocr is a recognizer trained for natural manga crops, not skeletonized masks. The second column is small, so after whitening it has too little visual context.

CV-only masks:

- `gray` and `lab` both find the text and produce 2 columns.
- `edge` / `grad` also find text but over-split into 3 columns because they emphasize outlines and disconnected stroke fragments.

### blue-vert(153s)

Corrected ROI: `(520,80,660,500)`, one long vertical column. Strong enough contrast and clean layout.

OCR:

- `raw`, `gray3`, `clahe`, `mask_bw`, `mask_bw_dilate`: all read `巻き戻せなかった`.
- `orig_on_white`: becomes `殺さ成せなかった`.

Why it is stable: this is a clean long column, so even binary mask retains enough glyph shape. But `orig_on_white` still proves that wiping background can distort model priors.

CV-only masks:

- `gray`, `lab`, `edge`, and `grad` all give vertical 1-column geometry.

### pink-vert(201s)

Corrected ROI: `(300,90,440,480)`, faint pink/low-contrast one-column text.

OCR:

- `raw`, `gray3`, `clahe`: correct `終わりにして`.
- `mask_bw`, `mask_bw_dilate`: `潔くり．．．`.
- `orig_on_white`: truncated to `終わりに`.

Why it fails: the mask captures only a small number of components (`n=2` in gray branch), so binarized text is too sparse and loses faint stroke detail. Raw crop keeps enough soft edges for manga-ocr.

CV-only masks:

- `gray` passes but with low text_likeness (`tl=0.5`).
- `lab`, `edge`, and `grad` improve geometry scores, but this does not mean their rendered masks are good OCR inputs.

### red-1col(42s)

ROI: `(1300,175,1420,340)`, small red diamond, expected text in a very small crop.

OCR:

- `raw`, `gray3`, `clahe`, `orig_on_white`: all read `それは、`.
- `mask_bw`, `mask_bw_dilate`: `．．．．．．`.

Why it fails: the crop is small and the glyph count is low. The binary mask removes too much visual information, so the recognizer falls back to punctuation-like outputs.

CV-only masks:

- `gray` passes vertical 1-column.
- `lab` and `edge` fail or no_text because there is only one useful component.
- `grad` passes but does not improve OCR.

### light-on-color(48s視感)

ROI: `(1520,360,1810,770)`, contains main text plus furigana/ruby-like small text. Current layout is `horizontal_ltr` bypass, not a clean vertical column.

OCR:

- `raw`, `gray3`, `clahe`: `視感が`.
- `mask_bw`: long dotted garbage plus `はい、`.
- `orig_on_white`: hallucinated phrase `その場合は、「感じる」`.

Why it fails: the problem is not OCR contrast. It is layout/ruby contamination. The mask contains multiple small components and side text, so binarizing amplifies junk structure and encourages hallucination.

CV-only masks:

- `gray` says horizontal.
- `lab` produces `unknown` with 3 columns.
- `edge` / `grad` exceed occupancy and include too much noise.

This belongs to the furigana/layout bucket, not OCR-image preprocessing.

### whitebox-2col(37s)

ROI: `(1300,230,1630,620)`, mixed glyph sizes: large `飽きた` plus smaller `もう自己顕示`. Current gray mask returns `layout=unknown`, splits into 3 columns, and the middle tiny crop reads `？`.

OCR:

- Real columns read fine when crop is meaningful: col0 `もう自己`, col2 `飽きた`.
- Middle over-split crop is junk regardless of raw/gray/CLAHE.
- `mask_bw` does not solve the split; it turns col0 into `もうｆｉ`, middle into tiny kana, col2 remains okay.

Why it fails: this is not an OCR-input problem. The crop list is wrong. The current x-center clustering over-splits mixed-size vertical text into an extra tiny middle column.

CV-only masks:

- `gray` / `lab`: still `unknown`, 3 columns.
- `edge` / `grad`: visually expose the correct two text columns, but also capture the white frame and character outlines. Geometry says vertical 2 columns, but occupancy is high (`occ=0.61-0.66`), so current scorer rejects it.

Research direction: edge/gradient may help GAP-C only if paired with border/long-line removal or component-level column proposal logic. Do not just switch to edge as the main mask.

### multiblock-whole(42s)

Whole-frame ROI with several separate text diamonds.

CV-only masks:

- All masks find many text-like components.
- `gray` gives 18 columns, `lab` 27, `edge` 14, `grad` 17.

Why it fails: preprocessing cannot solve multi-block. This needs upstream ROI splitting or a multi-block reject/segment step before columnizer.

### rain-no-text(69s)

Background rain texture, no text.

CV-only masks:

- `gray` / `lab` produce aligned-looking components, but occupancy is high and current gate rejects them.
- `edge` produces no useful ink, also reject.
- `grad` is very high occupancy and reject.

Why this matters: rain can align like text, so `text_likeness` alone is unsafe. Occupancy remains the correct rejection signal here.

## What Another Session Should Try Next

1. Keep manga-ocr input as `raw` crop by default.
2. Keep gray/Otsu as the primary CV mask.
3. Keep Lab only as fallback when gray fails; do not force it on corrected color ROIs.
4. Do not use `mask_bw` / binarized masks as OCR input except maybe as a rare fallback experiment with validation.
5. For GAP-C 37s, test edge/gradient only as a geometry helper:
   - remove long frame/border lines first,
   - ignore components touching ROI border,
   - use edge components to propose column groups,
   - still feed raw column crops to manga-ocr.
6. For furigana, solve layout/ruby filtering before OCR; contrast preprocessing is not the root cause.
7. For multi-block, solve ROI segmentation/reject before columnizer; no mask variant fixes whole-frame mixed text blocks.
